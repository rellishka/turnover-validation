# turnover-validation

Java/Spring Boot application supporting the tenant **turnover validation** business
process, built with ports and adapters (hexagonal architecture).

## Goal

Property management collects monthly turnover figures from tenants and must trust
them before they reach the Board's reports. This project shows that flow end to end,
with business rules kept in the domain:

- **Ingest** monthly turnover from an external Tenant App (REST) — durably, with
  retries, timeouts and an auditable `ImportRun` per run.
- **Validate**: flag a figure for review when it deviates from the previous month by
  more than 30%; an Asset Manager then **accepts** or **corrects** it, recording who,
  when and why (the audit trail that makes reports trustworthy).
- **Expose** only validated (`ACCEPTED` / `CORRECTED`) data to analytics consumers
  (e.g. Power BI) through a versioned, read-only SQL view.

The focus is not CRUD completeness, but showing where each concern belongs.

## Architecture

The app follows **ports and adapters** (hexagonal architecture) — see
[Hexagonal architecture: there are always two sides to every story](https://medium.com/ssense-tech/hexagonal-architecture-there-are-always-two-sides-to-every-story-bc0780ed7d9c).

- `adapter.in`
  - inbound HTTP adapter (controllers, request/response DTOs, mappers, exception handlers)
- `application`
  - `domain` — domain models, validation rules, `TurnoverManager`
  - `port.in` / `port.out` — inbound use cases and outbound ports
  - `service` — use-case orchestration and the transaction boundary
- `adapter.out`
  - `persistence` — JPA entities, Spring Data repositories, persistence mappers
  - `tenantapi` — outbound adapter for the external Tenant App (declarative client, retries)

The operational review UI is generated by **Apache Causeway** from the domain
actions (`accept` / `correct`), so there is no hand-written review controller — see
[docs/causeway-notes.md](docs/causeway-notes.md).

## Run

Build:

```bash
./gradlew build
```

Start (seeds demo data so the H2 console and analytics view show real rows):

```bash
./gradlew bootRun
```

Run tests:

```bash
./gradlew test
```

## Import endpoint

The manual counterpart to the scheduled import:

```bash
curl -X POST "http://localhost:8080/imports?period=2026-05"
```

It returns the `ImportRun` outcome (`SUCCESS` with an entry count, or `FAILED` with an
error message), and is idempotent — re-running a period does not duplicate data.

> Note: this demo points `tenant-api.base-url` at a placeholder Tenant App, so calling
> the endpoint returns a `FAILED` `ImportRun` (nothing to fetch). The sample data you
> see in the H2 console and the analytics view comes from the startup seed
> ([seed-data.sql](src/main/resources/seed-data.sql)), not from a live import.

## H2

The project uses H2 for local persistence.

H2 console:
- URL: [http://localhost:8080/h2-console](http://localhost:8080/h2-console)
- JDBC URL: `jdbc:h2:mem:turnovervalidation`
- User: `sa`
- Password: empty

The analytics view is created on startup; query it with:

```sql
SELECT * FROM analytics.v1_monthly_turnover;
```
